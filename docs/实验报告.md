## 1. 项目概述

本实验的主题是 **LoveLive! 综合管理系统**。系统采用 **B/S 架构**（Browser/Server，浏览器/服务器模式），前端为纯 HTML/CSS/JS 单页应用，后端基于 Node.js + Express，数据库使用 MySQL 8.0。系统将粉丝社区中两个原本割裂的需求——**动漫角色信息查询**与**舞团排练管理**——整合到一个 Web 应用中，用户通过浏览器访问即可使用全部功能。

选这个主题的有如下的需求的考量：LoveLive! 系列下有 7 个企划、9 个动漫团、67 个配音角色和 60 位声优，角色信息每次都要去专门的百科上去查，其实非常不方便；大多数粉丝表达自己对企划的喜爱也就是通过组成一些舞团，翻跳企划对应的舞蹈，有翻跳那必然就会有排练管理的需求。用 Excel 或群聊管理排练效率很低，角色信息还得去百科查。一个数据库驱动的 Web 系统正好解决这两个痛点。

系统拆成两个子模块，通过 `anime_group_id` 和 `character_id` 两条外键打通：

- **动漫角色介绍子系统**：企划 → 动漫团 → 角色 ← 声优（四张表，作品信息链）
- **舞台排练时间收集子系统**：舞团 → 舞见 / 排练 → 排练参与 ← 角色（四张表，排练管理链）

## 2. 运行与测试

### 2.1 启动方式

**前置条件**：Node.js 18+、MySQL 8.0+（需提前启动 MySQL 服务）。

**Step 1 — 克隆仓库**：

```bash
git clone https://github.com/KuuKangBest/Database_lovelive.git
cd Database_lovelive
```

**Step 2 — 配置数据库连接**：编辑项目根目录下的 `config.js`，修改为你的 MySQL 连接信息：

```js
module.exports = {
  db: {
    host: 'localhost',
    user: 'root',
    password: '你的密码',
    database: 'lovelive_db',
  },
  server: {
    port: 3000,
  },
};
```

**Step 3 — 安装依赖并启动**：

```bash
npm install          # 安装依赖（express、mysql2、cors）
npm start            # 一键启动（检查 MySQL → 初始化数据库 → 释放端口 → 启动服务）
```

`npm start` 会自动完成四步：连通 MySQL → 检查/初始化数据库 → 释放端口 3000 → 启动 Express 服务并打开浏览器访问 `http://localhost:3000`。

如需单独操作数据库：

```bash
npm run init-db      # 删库重建 + 分层导入全部数据
npm run seed         # 仅导入种子数据（不删表）
```

## 3. 需求分析

### 3.1 实体与属性

系统包含 8 个核心实体，分属两个子系统：

| 实体 | 中文名 | 所属子系统 | 核心属性 | 备注 |
|------|--------|-----------|----------|------|
| project | 企划 | 角色介绍 | project_id, name, description, start_date, end_date | name 唯一 |
| anime_group | 动漫团 | 角色介绍 | group_id, project_id, name, description, founding_date, disband_date | 外键 → project |
| cv | 声优 | 角色介绍 | cv_id, name, birth_date, agency | birth_date 用于按年龄筛选 |
| character | 角色 | 角色介绍 | character_id, group_id, cv_id, name, birthday, blood_type, height, hobby, cheering_color, eye_color, call_response | 外键 → anime_group, cv |
| dance_group | 舞团 | 排练收集 | dance_group_id, anime_group_id, name, description, created_date | 外键 → anime_group，**必选** |
| dancer | 舞见 | 排练收集 | dancer_id, dance_group_id, cn_name, qq, contact_info | 支持多团（通过 dancer_group_membership 关联表） |
| rehearsal | 排练 | 排练收集 | rehearsal_id, dance_group_id, rehearsal_date, start_time, end_time, location, max_participants, content_summary, status | status 支持 active/cancelled |
| rehearsal_participation | 排练参与 | 排练收集 | participation_id, rehearsal_id, dancer_id, character_id | 三元关联，(rehearsal_id, dancer_id) 唯一 |

实际实现中扩展了 3 张辅助表和 1 张关联表：

| 附加表 | 说明 |
|--------|------|
| dancer_group_membership | 舞见-舞团多对多关联，一人可隶属多个舞团 |
| character_relationship | 角色关系（姐妹、幼驯染、搭档等），支持关系图谱 |
| concert | 演唱会信息，支持时间轴展示 |
| song / performance | 曲目与演出（表已建，数据按需加载） |

值得注意的是，这三张辅助表其实就承担了该项目中对该企划的信息进行“百科”的功能，这一部分内容变动一般非常小，在实际项目中也就起到一个搜索的功能

### 3.3 级联策略

针对每个外键制定了不同的级联策略：

| 父实体 | 子实体 | 删除策略 | 理由 |
|--------|--------|----------|------|
| project | anime_group | CASCADE | 企划删了，下面团体没地方挂 |
| anime_group | character | CASCADE | 动漫团删了，角色也没归属 |
| cv | character | **SET NULL** | 声优删了，角色还在（换人配） |
| anime_group | dance_group | **SET NULL** | 动漫团删了，舞团不一定解散 |
| dance_group | dancer | **SET NULL** | 舞团解散，舞见可转到别的团 |
| dance_group | dancer_group_membership | CASCADE | 舞团删了，成员关联跟着删 |
| dancer | dancer_group_membership | CASCADE | 舞见删了，关联记录跟着删 |
| dance_group | rehearsal | CASCADE | 舞团解散，排练跟着删 |
| rehearsal | rehearsal_participation | CASCADE | 排练取消，参与记录跟着删 |
| dancer | rehearsal_participation | CASCADE | 舞见退出，参与记录清除 |
| character | rehearsal_participation | CASCADE | 角色删了，相关参与记录清除 |

与最初需求设计相比，最终 schema 中 `dancer.dance_group_id` 的删除策略从 CASCADE 改为了 **SET NULL**，这是因为在开发过程中引入了 `dancer_group_membership` 多对多关联表来支持舞见多团归属。舞团解散后舞见的主团字段置空，但舞见本身和她在其他团的关联记录都保留。业务层在删除舞见时会判断：如果只在一个团则彻底删除，如果在多个团则只从当前团移除（通过 `DELETE /api/dancers/:id/groups/:gid` 端点）。

### 5.2 范式分析

所有表满足 3NF（第三范式）：
- **第一范式**：所有字段原子化，无重复组。角色表把 birthday 拆成 MM-DD 字符串而非日期类型，是业务选择（只比较月日而非年份），不是违反 1NF。
- **第二范式**：所有非主属性完全函数依赖于主键。例如 rehearsal_participation 的三个外键都依赖于 participation_id，不存在部分依赖。
- **第三范式**：不存在传递依赖。例如 character 表中 group_id 决定 group_name，但 group_name 不放在 character 表里（通过 JOIN anime_group 获取）。

## 6. 物理设计：DDL 实现

数据库选型：**MySQL 8.0**，**InnoDB** 引擎，**utf8mb4** 字符集，utf8mb4_unicode_ci 排序规则。实际建表文件：`src/server/sql/schema.sql`，共 12 张表。

### 6.1 索引设计

| 索引目标 | 索引列 | 用途 |
|----------|--------|------|
| project.name | UNIQUE(name) | 企划名唯一约束 |
| anime_group.project_id | idx_group_project | 按企划查团体 |
| anime_group.name | idx_group_name | 按团名查 |
| cv.birth_date | idx_cv_birth | WHERE TIMESTAMPDIFF 年龄过滤 |
| character.group_id | idx_char_group | 按团体查角色 |
| character.name | idx_char_name | 按角色名查 |
| character.cv_id | idx_char_cv | 按声优查角色 |
| dance_group.anime_group_id | idx_dg_anime | 按动漫团查翻跳舞团 |
| dancer.dance_group_id | idx_dancer_group | 按舞团查舞见 |
| dancer.qq | idx_dancer_qq | QQ 号搜索 |
| rehearsal.rehearsal_date | idx_reh_date | 日期范围查询 |
| rehearsal.dance_group_id | idx_reh_group | 按舞团查排练 |
| rehearsal_participation | idx_rp_rehearsal, idx_rp_dancer, idx_rp_character | JOIN 与统计常用 |
| dancer_group_membership | idx_dgm_dancer, idx_dgm_group | 多团查询 |

### 6.2 约束与默认值

- UNIQUE(rehearsal_id, dancer_id)：分身乏术：一次排练，一个舞见只能参与一次（扮演一个角色）
- UNIQUE(dancer_id, dance_group_id)：舞见不能重复加入同一舞团
- UNIQUE(character_id_1, character_id_2, relation_type)：两个角色之间同类型关系不重复
- max_participants DEFAULT 0：0 表示不限人数
- status DEFAULT 'active'：新建排练默认有效
- rehearsal 的 excluded_chars / extra_chars 字段用 TEXT 存逗号分隔的 ID 列表，支持灵活的角色槽位管理（排除不需要的角色，或添加跨团角色）

### 6.3 字符集与引擎

所有表统一使用 InnoDB 引擎。InnoDB 支持事务（排练创建使用显式事务保证一致性）、行级锁、外键约束。字符集选择 utf8mb4 是因为日语角色名和声优名可能包含 4 字节 Unicode 字符。

## 7. 视图与触发器

### 7.1 rehearsal_with_count 视图

这是实现中最重要的视图，聚合排练的当前参与人数和饱和度：

```sql
CREATE VIEW rehearsal_with_count AS
SELECT
    r.*,
    COUNT(rp.participation_id) AS current_participants,
    CASE
        WHEN r.max_participants = 0 THEN NULL
        ELSE ROUND(COUNT(rp.participation_id) / r.max_participants * 100, 1)
    END AS occupancy_rate,
    CASE
        WHEN r.max_participants = 0 THEN 'unlimited'
        WHEN COUNT(rp.participation_id) >= r.max_participants THEN 'full'
        WHEN COUNT(rp.participation_id) / r.max_participants >= 0.8 THEN 'near_full'
        ELSE 'available'
    END AS occupancy_status
FROM rehearsal r
LEFT JOIN rehearsal_participation rp ON r.rehearsal_id = rp.rehearsal_id
GROUP BY r.rehearsal_id;
```

四个饱和度等级在前端用不同颜色展示：unlimited（灰）、available（绿）、near_full（橙）、full（红）。

### 7.2 四个触发器

**trg_rehearsal_check_max**（BEFORE INSERT on rehearsal）：`max_participants < 0` 时 SIGNAL 拒绝。

**trg_rehearsal_update_check**（BEFORE UPDATE on rehearsal）：修改上限时检查（1）不能为负数；（2）新上限不能小于当前参与人数。

**trg_participation_check_full**（BEFORE INSERT on rehearsal_participation）：当 `max_participants > 0` 且 `current >= max_participants` 时拒绝插入。

**trg_participation_check_conflict**（BEFORE INSERT on rehearsal_participation）：检查同一舞见在同一天是否有时间段重叠的其他排练。重叠条件：

```
r1.rehearsal_date = r2.rehearsal_date
AND r1.start_time < r2.end_time
AND r2.start_time < r1.end_time
```

所有触发器使用 `SIGNAL SQLSTATE '45000'` 抛出自定义错误信息，前端可以解析并展示。

### 8.3 语义搜索模块

这里其实做的不是很仔细，只是让ai进行了一些较为基础的实现，比如在角色搜索输入“钢琴”会显示一些喜好钢琴的动漫角色，输入“千”会显示名字跟千相关的角色（但是做的并不是很细，后续如果真的能做成比较成熟的项目在这里肯定是需要做优化的）



`src/server/semantic-search.js` 实现了一套**纯 JavaScript 的 TF-IDF + Cosine Similarity 搜索引擎**，零外部 NLP 依赖：

1. **分词**：中文 bigram（2-gram）+ 单字过滤，支持中日混合文本
2. **同义词扩展**：内置 18 组同义词映射（如"钢琴"↔"音乐""演奏"，"绘画"↔"画画""设计"）
3. **TF-IDF**：对 67 个角色的 name + hobby + description 建立倒排索引，IDF 平滑：ln((N+1)/(df+1)) + 1
4. **查询增强**：原始查询词的权重 ×3，扩展同义词权重 ×1
5. **向量检索**：预计算所有角色的 L2 归一化 TF-IDF 向量，查询时 cos 相似度排序
6. **结果**：返回 topK 结果（默认 20），带上 similarity 分数

搜索范围覆盖角色名、爱好、描述、声优名、团体名、瞳色、应援色、互动词和血型，多个查询词之间的关系是 AND（每个词都要命中）。

## 9. 前端实现

前端是一个纯手写的单页应用（SPA），由三个文件组成：`index.html`（页面结构）、`app.js`（~1100 行业务逻辑）、`style.css`（样式）。没有引入任何前端框架或构建工具，所有 DOM 操作、事件绑定、异步请求、Canvas 绘图均用原生 API。页面间切换通过 `showPage(name, filter)` 函数控制 CSS class `active` 的显隐，filter 参数支持跨页面传参（例如从企划卡片点击进入该企划下的团体列表）。

- **角色偏见卡片选择器**（`buildCharBiasCards`）：按动漫团分组展示所有角色的 mini 卡片（含应援色背景），可多选。选中后调用 `GET /api/dancers/by-character?q=角色名1,角色名2` 后端接口，反查历史上扮演过这些角色的舞见。选中的角色卡片高亮显示其应援色。这是"角色偏见"功能的体现——用户可以说"我想找演过高坂穗乃果的舞见"。

**列表展示**：舞见按所属舞团分组（支持多团——同一舞见出现在多个团下时在各团各显示一张卡片，带"+N团"标识）。每张卡片含：
- CN 名
- QQ 号标签
- 角色偏好（出演次数最多的角色名）
- 卡片边框色 = 偏好角色的应援色
- 编辑按钮（跳转编辑弹窗）、移除按钮（区分"从当前团移除"/"彻底删除"）

**舞见编辑弹窗**：CN 名 + QQ 号（纯数字校验）+ 所属舞团下拉框。支持修改主团：如果修改后的主团与旧主团不同，旧团自动保留在 `dancer_group_membership` 中，实现无感的多团迁移。

**舞见详情弹窗**（`showDancerDetail`）：
- 头像占位（CN 名首字）+ CN 名
- 所属舞团标签列表（主团 + 关联团），点击可跳转舞团详情
- **出演角色统计**：按扮演次数降序排列的角色卡片，卡片顶部色条为角色应援色，点击可跳转角色详情。数据来自 `GET /api/dancers/:id/stats`（GROUP BY character_id + COUNT）。
- **全部排练记录**：该舞见参与的所有排练，每条列出角色名（带应援色圆点）+ 舞团名 + 日期 + 时间段 + 地点 + 内容摘要。已取消排练灰色标注。按日期倒序排列。数据来自 `GET /api/dancers/:id/rehearsals`。

### 9.8 关系图谱

角色详情弹窗中点击"关系图谱"按钮，从 `GET /api/characters/:id/relationships` 加载该角色同团成员之间的关系数据，在 Canvas 上绘制力导向图。节点大小区分中心角色（34px，带虚线外环）和普通角色（24px），节点颜色为角色应援色。8 种关系类型用不同颜色区分：姐妹（#e91e63）、幼驯染（#ff9800）、搭档（#4caf50）、憧憬（#9c27b0）、挚友（#2196f3）、对手（#f44336）、同学（#607d8b）、师徒（#00bcd4）。关系边上的标签带白色背景，单向关系（憧憬、对手）只画单向箭头，其余双向。节点可拖拽，点击其他角色节点可跳转该角色的详情。

### 9.9 演唱会时间轴

在企划页或团体页中点击"演唱会时间轴"按钮，弹出表格式时间轴。每行一场演唱会，含彩色标签（按巡次/类型分色）、日期、演唱会名称。不同巡次的标签颜色各异：1巡绿、2巡蓝、3巡紫、4巡橙、5巡红、Final 深红等。数据来自 `GET /api/concerts?project_id=...`。
